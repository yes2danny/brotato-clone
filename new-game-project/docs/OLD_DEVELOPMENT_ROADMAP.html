How to read this schedule

Days 1–2 assume you can pour 6–8 focused hours per day (build + test + commit). Day 3 onward assumes a job day plus ~1.5–2 hours most evenings — tasks are scoped to finish one vertical slice or one bug class, not “rewrite AI.” If a box slips, move it forward; do not collapse multiple day-sized blocks into one tired night.

Ship in small commits: “upgrade UI wired,” “shop opens after wave,” “XP resets on restart.” That keeps morale and bisects regressions.

Intensive block — days 1 & 2

Dense checklist while you have time; reorder within a day if you prefer UI-first, but do not skip run-safety fixes.

Day 1 — Run safety + level-up UX Full day

Morning — unblock soft-locks

  • Add XPSystem.reset_run() (or equivalent): zero XP, level, thresholds; call from GameManager.restart_game() and optionally from Main._ready() for clean boots.
  • Playtest: level up twice, restart from game over, confirm XP bar matches level 1.
  • Confirm WeaponController.bullet_scene assigned on Player in saved scene — document in commit message if you fix it.

Afternoon — upgrade picker (minimum viable)

  • In UpgradeManager scene: add four Button nodes (or one GridContainer of buttons); set text in show_upgrades() from UPGRADE_NAMES.
  • Connect each button to _on_upgrade_chosen(0..3); hide overlay and get_tree().paused = false on pick.
  • Optional quality: number keys 1–4 as alternate input; debounce double-press.
  • Visual pass: semi-transparent full-screen tint + panel so the player understands the game is waiting for a choice.

Evening buffer — polish or stop

  • One playthrough to wave 5+ with intentional level-ups; fix any pause/upgrade edge case.
  • If time remains: stub res://scenes/MainMenu.tscn (title + Play + Quit) and point project main scene OR only load_main_menu() — avoid broken pause quit.
Day 2 — Shop in the wave loop + first data Full day

Morning — wire economy to waves

  • Design decision (pick one, document in code comment): shop opens immediately after wave end, or after the short break. Brotato-likes usually shop then next wave.
  • From WaveManager._end_wave() (or after break before _start_wave), call ShopManager.open_shop(); ensure wave counter / spawner state stays coherent.
  • Gold: on GameManager.register_kill() or enemy death hook, ShopManager.add_gold(small_amount); tune starting gold in inspector.

Afternoon — ShopUI that actually sells

  • Build panel: gold label, 4 slots (icon placeholder + name + price + Buy), Leave button → close_shop().
  • Populate from shop_opened signal; gray out / disable bought slots.
  • Create 3–5 ItemData .tres files (even with duplicate art): e.g. +HP, +speed, +damage, +pickup radius (wire radius into XPGem attract if not global yet).
  • Assign available_items on ShopManager in Main.tscn.

Evening — weapon purchase stub or defer

  • Either: implement WeaponController.apply_weapon_data(WeaponData) and call from ShopManager._apply_to_player, or hide weapon entries in shop until Day 4 — but do not leave “buy” printing only.
  • End-to-end test: two waves, two shop visits, gold increases, purchases change stats.

Week 1 — post-work evenings (days 3–7)

~1.5–2h per night; one primary outcome per day.

Day 3 — Main menu + boot flow Evening
  • Finish or polish MainMenu.tscn: Play loads Main.tscn, Quit exits.
  • Set project run/main_scene to menu if you want proper boot order.
  • From game over, Restart vs Menu both reset autoloads consistently.
Day 4 — WeaponData runtime path Evening
  • WeaponController: optional @export var weapon_data: WeaponData; in _ready, apply damage, fire rate, bullet speed, range from resource.
  • Create 2 WeaponData resources (e.g. fast weak vs slow strong); second weapon only acquirable via shop for now.
  • Fix projectile count / spread if you touch firing logic — keep scope tiny.
Day 5 — Pause stack audit Evening
  • Reproduce: pause during shop, during upgrade, during game over — ensure only one system owns paused or use layered pause flags.
  • Decide: block ESC during upgrade/shop or route to “confirm leave” — document behavior.
  • Fix worst bug found; defer nice-to-have.
Day 6 — WaveData step 1 Evening
  • Create wave_01.tres … wave_03.tres with different spawn_interval / duration labels.
  • WaveManager: @export var waves: Array[WaveData]; index by current_wave - 1 with fallback to defaults if out of range.
  • Spawner: optional per-wave override of interval min/floor — even one field wired counts as win.
Day 7 — HUD + feedback pass Evening
  • GameUI: wave number label listening to WaveManager signals; gold label listening to gold_changed.
  • Light juice: hit flash on enemies or floating damage numbers (pick one, keep under ~40 new lines).

Week 2 — days 8–14

Still evening-sized; bias toward “more content through data” instead of new engines.

Day 8 — Enemy variety via data Evening
  • Audit EnemyData usage; if unused, hook one stat (speed or HP) from resource on enemy _ready.
  • Duplicate one enemy scene with different tint + data; add to spawner pool for high waves only (simple wave threshold in spawner).
Day 9 — Boss or elite wave stub Evening
  • Use WaveData.is_boss_wave: on that wave, spawn one oversized enemy or reduce count + buff HP.
  • Boss drops extra gold or large XP gem — one reward type.
Day 10 — Items that need code hooks Evening
  • Implement bonus_armor in HealthSystem (simple percent reduction cap).
  • Global pickup radius: player stat modified by item, read by XPGem via group lookup or signal.
Day 11 — Audio pass 1 Evening
  • Add AudioStreamPlayer2D for fire, hit, gem, wave start (CC0 or your own).
  • Autoload optional music bus with ducking when shop opens — skip if over scope; SFX alone is fine.
Day 12 — Balance spreadsheet night Evening
  • Export or list key tunables: spawn_interval curve, XP thresholds, gold per kill, shop prices.
  • Play one run, note time-to-first-death and boredom points; adjust two numbers only.
Day 13 — Second map swap experiment Evening
  • Instance room_cemetery.tscn instead of crypt in a duplicate test scene OR export var map on Main.
  • Verify MapBoundary limits match; fix camera if scale differs.
Day 14 — Playtest build + backlog grooming Evening
  • Export Windows build; write five bullet “known issues” from one friend or self with fresh eyes.
  • Reorder next month backlog: mark shipped vs cut vs “month 2.”

Months 2–5 — expansion skeleton

Roughly one month per column; overlap is normal. Expand each into your own issue tracker when ready.

Month 2 Month 3 Month 4 Month 5
Character stats screen; 2–3 playable “classes” as different starting WeaponData + stat offsets. Weapon fusion or duplicate-item stacking rules; more projectiles (shotgun pattern) using existing projectile_count. Elite modifiers on enemies; status effects (burn/slow) if you want ARPG depth. Difficulty tiers; endless mode toggle; final boss sequence for “win” state.
Meta currency + persistent unlocks (new items appear in pool after unlock). 20+ items and 6+ weapons as data-only drops; rarity weights in shop roll. Save/load run (mid-run quit) OR explicit roguelike “no save” policy polished in UI copy. Performance: object pooling for bullets and gems; profiling on low-end GPU.
Localization hooks for all player-facing strings. Controller support and aim-assist options. Steam / itch page assets: trailer, GIFs, capsule art. Release candidate: bug bash week, version numbering, patch notes template.

Five months to “done” is realistic for a solo Brotato-scale hobby vertical slice if you protect the spine (menu → run → meta → repeat) and add breadth mostly through .tres + scenes. A commercial clone with full content parity takes longer; adjust the table to your actual hours per week.
