20-wave roadmap, with math. [v3 — difficulty rebalance]

v3 rebalance pass: the v2 HP/DMG multipliers started at 1.00× making wave 1 enemies die in 1–2 hits — trivially easy. 1 — What's broken / weak in the v1 plan

The v1 doc has the right shape, but several gaps would break tuning the moment you start playtesting. Reading it carefully:

Fault A — No numbers per wave (NPW). v1 says “relaxed interval,” “pressure up,” “hard floor on interval.” That's vibes, not data. You can't tune what you can't count. Wave 1 should spawn N enemies total, with max M alive at once — those numbers don't exist anywhere.
Fix: §3 below defines N_total(w) and N_max(w) as equations, then §4 prints the resulting integer per wave.
Fault B — Enemy HP scales but player damage doesn't. v1 has health_scale_per_wave growing 4–8% per wave, but nothing about player DPS keeping up. If guns stay the same and HP doubles, the player gets slower at clearing without it feeling like power growth — they just feel weaker.
Fix: §7 defines a player DPS budget that targets clearing the concurrent-cap in ~5s, so weapon/damage upgrades have a target to hit per shop visit.
Fault C — Mini-boss is one row, not a schedule. v1 mentions “mini at 5/10/15, boss every 10.” That's a sentence, not a design — what scales between minis? Why does wave 15 mini differ from wave 5? What HP, what damage, what reward?
Fix: §6 specifies count, HP multiplier, damage multiplier, reward delta, and a unique twist per mini-boss wave.
Fault D — Tier bands name sprites, but no wave-by-wave unlock. v1 lists Tier F → Tier S but never says “Sorcerer_LVL2 first appears wave 9.” Without a schedule the spawner has nothing to read.
Fix: §5 prints all 54 characters with their unlock wave and tier; §4's table shows which pool is active per wave.
Fault E — Boss wave 20 = “FrogBoss with ×5–8 HP.” That's a fat trash mob, not a boss. A real boss needs phases, telegraphed attacks, or adds — otherwise the run ends with one big HP sponge.
Fix: §6 splits boss into three phases (100/66/33% HP), each unlocking a new attack and an adds wave from the existing megapack.
Fault F — Spawn interval is the only difficulty knob. v1 says “interval −0.1.” But two enemies at 0.4s intervals with HP 50 each is very different from one big enemy every 2s with HP 500. v1 conflates pressure with difficulty.
Fix: §3 separates four independent levers — spawn rate, HP mult, damage mult, and pool composition — so you can change pressure without changing lethality, or vice versa.
Fault G — No win condition. Both v1 docs end at “wave 20 boss, ×5–8 HP.” Nothing says what happens when you kill it. Does the run end? Roll into endless? Show a victory screen? Without that decision the rest of the boss design has nothing to anchor to.
Fix: §6 specifies: boss death → 3-second cinematic pause → GameManager.win_run() → results screen → meta currency awarded → menu. Endless mode is a separate toggle, not the default. Repo status: win_run() exists on GameManager (cleanup + placeholder meta + victory UI); slow-mo / full results flow still optional polish.

3 — The math (the four difficulty levers)

Four independent equations. Each takes the wave number w (1 → 20). You can tune each curve without touching the others.

3.1 — Wave duration

How long the player has to survive in a wave (seconds).

D(w) = 20 + 2·w # seconds → w1=22 w5=30 w10=40 w15=50 w20=60

Matches Brotato's 20→60s ramp. Long enough at wave 20 to give the boss room to breathe; short enough at wave 1 not to bore.

3.2 — Spawn interval (pressure lever)

Seconds between trash spawns from the ring spawner. Floor tightened to 0.35s. [v3 change] Starting interval reduced from 2.0 to 1.60 so wave 1 already has real spawn pressure — the old formula gave 1.92s which meant the player had nearly 2 full seconds between each enemy. Floor hits at wave 15–16.

T(w) = max(0.35, 1.60 − 0.08·w) # seconds [rebalanced v3] → w1=1.52 w5=1.20 w10=0.80 w15=0.40 w16+=0.35 (floor) OLD (v2, too gentle): T(w) = max(0.4, 2.0 − 0.08·w) → w1=1.92 w5=1.60 w10=1.20 w15=0.80 w20=0.40

3.3 — Total spawns per wave (the NPW you asked for)

Total enemies that will spawn over the wave. We use a smoother direct curve so wave 20 isn't bottlenecked by the floor of T:

N_total(w) = round(10 + 4·w + 0.5·w²) → w1=14 w2=20 w3=26 w4=34 w5=42 w6=52 w7=62 w8=74 w9=86 w10=100 w11=114 w12=130 w13=146 w14=164 w15=182 w16=202 w17=222 w18=244 w19=266 w20=290*

* Wave 20 is a boss wave override — it does not follow this formula. Trash is throttled (T=1.0s, N_max=30) so the boss reads clearly. ~60 trash spawns + ~13 scripted boss adds + 1 boss ≈ 75 total enemies. See §6.

3.4 — Max concurrent enemies (the screen-pressure lever)

Hard cap on enemies alive at once so the player isn't drowned and performance doesn't tank on low-end PCs. [v3 change] Base raised from 10 to 16 and linear coefficient from 2→2.5 so wave 1 opens with 19 slots instead of 12 — more enemies on screen at once from the jump.

N_max(w) = round(16 + 2.5·w + 0.18·w²) [rebalanced v3] → w1=19 w5=33 w10=59 w15=94 w16+=floor (boss wave 20 overrides to 30) OLD (v2): N_max(w) = round(10 + 2·w + 0.15·w²) → w1=12 w5=24 w10=45 w15=74 w20=110

If get_tree().get_nodes_in_group("enemies").size() >= N_max(w), spawner skips its tick. This is your kindness valve.

3.5 — Enemy HP multiplier

Multiplies each enemy's base HP (from its scene's max_health export). [v3 change — this was the main problem.] The old formula started at 1.00× meaning wave 1 enemies had literally zero HP scaling. A GoblinGreen (30 base HP) at 1.00× dies in ~1.5 seconds at starting DPS. Now we start at 1.80× so wave 1 enemies take 2–3 hits and actually threaten the player. The curve grows at 0.16 per wave (vs old 0.08) for a steeper but fair ramp.

HP_mult(w) = min(5.0, 1.80 + 0.16·(w − 1)) [rebalanced v3] → w1=1.80 w5=2.44 w10=3.24 w15=4.04 w20=4.84 (cap of 5.0 not reached until wave 21 — no artificial ceiling in the run) OLD (v2, the reason early waves felt trivial): HP_mult(w) = min(3.5, 1 + 0.08·(w − 1)) → w1=1.00 w5=1.32 w10=1.72 w15=2.12 w20=2.52

3.6 — Enemy damage multiplier

Multiplies contact damage on enemies. Still grows slower than HP (so fights get longer before they get more lethal), but starts at 1.15× instead of 1.00× so a GoblinGreen hit on wave 1 deals 11.5 instead of 10 — small but noticeable. Cap raised to 2.8. Hits cap around wave 19.

DMG_mult(w) = min(2.8, 1.15 + 0.09·(w − 1)) [rebalanced v3] → w1=1.15 w5=1.51 w10=1.96 w15=2.41 w19=2.77 w20=2.80 (cap) OLD (v2): DMG_mult(w) = min(2.0, 1 + 0.04·(w − 1)) → w1=1.00 w5=1.16 w10=1.36 w15=1.56 w20=1.76

3.7 — XP per kill

XP_per_kill(w) = round(5 + 0.8·w) → w1=6 w10=13 w20=21

3.8 — Gold per kill

Gold_per_kill(w) = max(1, round(1 + 0.25·w)) → w1=1 w5=2 w10=4 w15=5 w20=6

Wave 19 total gold ≈ 266 × 6 = 1596. Across the whole run a typical player banks ~5–7k gold before the boss intro — enough to fully kit out the final shop visit.

3.9 — Player DPS budget (target, not enforced)

For the player to feel powerful, they should be able to clear N_max(w) enemies in roughly 5 seconds. So the required DPS is:

DPS_target(w) = (N_max(w) · HP_avg_pool(w)) / 5 HP_avg_pool(w) ≈ HP_mult(w) · base_avg(w) base_avg(w) grows from 15 (w1 trash) → 80 (w20 trash) as pool shifts

Concrete: wave 10 ≈ (45 · 1.72 · 40) / 5 = 619 DPS. Wave 20 ≈ (110 · 2.52 · 80) / 5 ≈ 4435 DPS. Your default weapon at 10 dmg × 4 shots/s = 40 DPS, so the player needs roughly ×100 effective DPS growth across the run. That's what items and weapon upgrades pay for — see §7.

6 — Mini-boss schedule + final boss design

6.1 — Mini-boss generic rules

  • Spawn at wave start, before trash ramp begins (1s delay).
  • add_to_group("mini_boss") so UI can show a skull icon and a separate HP bar.
  • Visual: scale = Vector2(1.15, 1.15) + slight red modulate to read as "elite."
  • Death reward: 3× gold, drops a guaranteed large XP gem worth 5× normal kill XP.
  • Mini-bosses do not count toward N_total; trash still spawns normally around them.

6.3 — Final boss (wave 20)

Wave 20 is the entire run's payoff. Trash spawns are throttled (T = 1.0s instead of 0.4s) and N_max is reduced to 30 so the boss is readable.

Boss: FrogBoss, base HP 2500, scaled to ~6300 HP at wave 20 multiplier.

  • Phase 1 (100% → 66% HP) — Slow chase. Periodic tongue lash attack (single-line telegraph, 1.2s windup). Trash pool: tier D + E only.
  • Phase 2 (66% → 33% HP) — Adds wave on transition: spawn 8 × Goblin_Barrel red (suicide bombers) all at once. Boss gains "leap" attack — short telegraph circle, deals high damage on landing.
  • Phase 3 (33% → 0% HP) — Adds wave: spawn 1 × GameMaster + 4 × Mushroom (poison cloud variant — future). Boss now also fires a 5-projectile spread on a 4s cooldown.
  • Death — All remaining enemies die, 3-second slow-mo, GameManager.win_run(), results screen, run ends.

Coding shortcut: phase transitions can be three booleans on the boss script + a signal each time HP crosses a threshold. You do NOT need a state machine. Yet.

7 — Player power curve (so guns scale with enemies)

If you don't scale the player, the curve from §3.5–3.6 means the run gets unwinnable around wave 12. Three independent player levers:

7.1 — Weapon tiers (shop unlocks)

Wave unlockedTierBase DPSExample name
1 (default)I — Pistol40Rusted Revolver

Each tier is roughly ×2.5–3 the previous. Player carries 1–2 weapons; the ramp from DPS_target in §3.9 is met if they buy roughly every other tier on schedule.

7.2 — Item-stat multipliers

Items multiply weapon DPS via 4 stats — these get aggregated into the equation below:

effective_DPS = weapon_DPS × (1 + damage_pct) × (1 + atk_speed_pct) × (1 + crit_chance · crit_mult) × projectile_count

By wave 20 a typical run should hit roughly +150% damage, +80% atk speed, 30% crit @ 2× mult, 2 projectiles: 5000 × 2.5 × 1.8 × 1.6 × 2 ≈ 72000 DPS peak — well above the 4435 DPS_target from §3.9, leaving headroom for unlucky runs.

7.3 — Survivability scaling

Enemy DMG multiplier hits 1.76× at wave 20. Player max HP should grow to ~3× via items + level-ups, so they take real damage but aren't one-shot.

target_player_HP(w) ≈ 100 · (1 + 0.10·w) → w1=110 w10=200 w20=300

8 — Build order (small scope, ship-able chunks)

Don't try to land this whole doc in one weekend. Five chunks, each one playable on its own:

Chunk 1 — Spine (1 evening)

  • Add @export var waves: Array[WaveData] to WaveManager.
  • _start_wave() reads waves[current_wave - 1], applies wave_duration, spawn_interval, calls spawner.apply_wave(wave_data).
  • EnemySpawner.apply_wave() replaces its enemy_scenes with the wave's pool.
  • Create wave_01.tres through wave_05.tres with numbers from §4. Game should now play differently per wave.

Chunk 2 — Scaling hook (1 evening)

  • Add EnemyScaler.apply(enemy, wave) static utility — multiplies HP and damage from EnemyData by the curves in §3.5 / §3.6.
  • Call from EnemySpawner._spawn_enemy() right after instantiate().
  • Cap N_max: spawner skips its tick if get_tree().get_nodes_in_group("enemies").size() >= N_max(current_wave).

Chunk 3 — Roster + tier scenes (3–4 evenings)

  • Write a one-off editor script that walks assets/sprites/characters/megapack/ and emits one EnemyData.tres + one Enemy_<name>.tscn per folder, using tier base stats from §5.
  • Hand-assign per-tier stats (HP, speed, damage) on the EnemyData; sprite paths come from the folder.
  • Fill wave_06.tres through wave_20.tres using §4.

Chunk 4 — Mini-boss channel (1 evening)

  • Extend WaveData: mini_boss_scene: PackedScene, mini_boss_count: int, mini_boss_hp_mult: float, mini_boss_dmg_mult: float.
  • On wave_started, if mini_boss_scene != null, spawn at start with multipliers, add to "mini_boss" group.
  • Wave 15 "shield link" can be a future polish pass — ship the simple version first.

Chunk 5 — Final boss (2–3 evenings)

  • New scene Enemy_FrogBoss.tscn with a dedicated FrogBossAI.gd (don't reuse EnemyAI here — boss earns its own script).
  • Three phase booleans + HP-threshold signals. Phase 1 = wired AI. Phases 2 + 3 add one attack each.
  • GameManager.win_run() + simple results screen. Endless mode is a flag on GameManager, default off.

Chunk 6 — Player power (parallel anytime)

  • Author WeaponData resources for tiers I → VI per §7.1.
  • Shop rolls weighted by current wave — only show appropriate tier or one below.
  • Item resources for the four stat buckets in §7.2.

Order matters: chunks 1 + 2 alone make the game feel like it has waves. Don't author all 54 scenes before that spine works — you'll re-do half of them when you realize the scaling needs a different export.

9 — What I'm NOT putting in this doc on purpose

  • Difficulty modes / danger levels. Brotato has 5 of these. We don't need them for v1 — ship the default curve, add a "Hard" toggle later that just multiplies HP/DMG by 1.25.
  • Status effects (burn, slow, poison). Tempting, but they're a multi-week system. Skip until after first playtest.
  • Enemy abilities beyond contact damage. Charges, ranged shots, leaps — those are tier-by-tier polish. The math above works with everyone running at the player.
  • Procedural wave generation. 20 hand-authored WaveData.tres files. No PCG. You'll thank yourself when tuning.
